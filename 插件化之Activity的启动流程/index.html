Binder通信模型

Binder框架定义了四个角色：
用户空间
Server：定义Binder
Client：获取Binder的引用
ServeiceManager:将字符形式的Binder名字转化成对Binder的引用。
内核空间
Binder驱动：工作于内核态，以字符设备中misc设备注册在/dev下，用户通过/dev/binder访问，驱动负责进程之间Binder通信的建立，Binder在进程之间的传递。

ServiceManager的启动

Binder框架中四个角色的关系如下，ServiceManager是Binder机制的守护进程。接下来简单的看一下ServiceManager的启动过程。

  1. main函数

ServiceManager在用户空间的代码位于frameworks/base/cmds/servicemanager

Linux 字符设备驱动
file_operations:

该结构是将系统调用 和驱动程序连接起来，这个结构的每一成员都对应着一个系统调用。当用户进程利用系统调用对设备进行读写操作的时候，这些系统调用通过设备节点中的主设备号和次设备号来确定相应的驱动程序，而每一个字符驱动在linux内核中又是由cdev结构体来描述的，其中cdev结构体中含有成员fops结构体，然后就可以读取file_operations结构体中相应的函数指针，接着把控制权交给函数，从而完成linux设备驱动程序的工作。

对应binder驱动来说，上述函数对应的实现分别是
binder_open,binder_ioctl，binder_mmap,binder_release,binder_flush。

具体到servicemanager中，fd = open(“/dev/binder,O_RDWR”);
通过系统调用open()打开设备文件，此设备节点对应有一个设备号。打开/dev/binder时，通过主次设备号找到相应的字符驱动程序，即在cdev链表中找到cdev这个结构体，cdev里面包含file_operations结构体，含有对设备的各种操作，打开时即调用里面的open函数指针指向的open函数，也就是驱动中的bind_open函数。

  1. 注意此处open(“/dev/binder”,0_RDWR)是一次系统调用，最终会在内核空间执行binder驱动的binder_open(struct file *filp)函数。
  2. mmap的调用最终会在内核控件执行binder驱动的binder_mmap()函数，这个函数在内核空间创建了一个一个mapsize大小的缓冲区，并将/dev/binder设备文件映射到这块内存上。这里同时使用进程虚拟地址空间和内核虚拟地址空间来映射同一个物理页面，当client需要和server进行通信时，只需要将client进程的数据拷贝进内核空间的缓冲区，由于这个缓冲区和server进程共享同一块物理页，server进程可以直接获取client的数据，不需要再从内核空间拷贝到server进程空间。

ActivityManager的调用相关类

ActivityManagerNative是对外的接口类，所有对AmS的调用都通过它来进行，其内部维护了一个IActivityManager的单例，如果对AmS的调用发生在system_process进程，那么单例返回ActivityManagerService，如果调用发生在其他进程，则返回ActivityManagerProxy，此时获取的是ActivityManager的远程接口。

ActivityThread是应用进程的入口类，其中包含应用进程的main函数，在进程启动后，ActivityThread开启事件循环，并创建了一个H类的实例来处理异步操作。
如上图所示，ActivityThread内部包含一个ApplicationThread类，这个类的主要作用是用来和AmS进行通信。

通过ActivityManagerNative获取AmS的远程接口是通过名字获取的，这是一次实名Binder调用，那么AmS如何回调ActivityThread进程中的方法呢，这里是通过匿名Binder，通过AmS的远程接口将ActivityThread中的Binder传递给AmS，我们来看一下具体的过程。

  1. 创建ApplicationThread

在Ams中，创建一个新的Activity并不是一件容易的事情，其中涉及到AmS对进程的管理，对Activity的管理等等诸多方面，我们不做太多的分析，简单看一下流程。

Application的创建流程

  1. AmS从zygote fork一个虚拟机进程，执行ActivityThread的main函数。
  2. 执行ActivityThread的attach函数，调用AmS的attachApplication函数。
  3. 在AmS中处理一些和进程相关的内容，如果当前App没有执行过dexopt，那么先执行dexopt
  4. 回调ApplicationThreadNative的bindApplication函数
  5. 执行ActivityThread的handleBindApplication函数

总结一下

  1. Application的实例化和生命周期回调都有Instrumentation执行
  2. Application的onCreate方法之前，先执行attachBaseContext方法，此时将ContextImpl的实例赋值给mBase变量，此后所有对context的调用都被委托给mBase变量
  3. 应用的ClassLoader是PathClassLoader，父加载器是BootClassLaoder
  4. MultiDex的实现方式是在attachBaseContext时，修改应用的ClassLoader的pathList，这个过程中需要调用makeDexElement，内部的实现会触发dexopt。由于整个过程是在主线程同步执行的，所以会出现黑屏和卡顿。
